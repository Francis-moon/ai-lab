# 目标:实现简化版 Parking Semantic Map

最小对象集先做 Zone / Lane / Slot，地图从导航工具升级为业务数据库。
也和 BP 里“统一空间基准、空间智能体、动态孪生、协同调度”的方向一致。

这周的重点不是“画地图”，而是：
让事件、任务、状态都落到空间对象上。

## 这一版你会得到什么

得到一个最小的“语义地图系统”：
Zone：区域
Lane：通道
Slot：车位

并且支持：
车位属于哪个通道
通道属于哪个区域
事件发生在某个对象上
系统可以查询：
某个区域有多少空车位
某条通道下有哪些车位
某个车位属于哪个区域
事件仍然会触发：
状态更新
任务生成
任务执行
结果回写
这就是“语义地图最小原型”。

## 本周必须真正理解的3句话

1.车位不是编号，而是空间实体。
这正是你们战略文档里强调的关键原则。

2.识别结果必须落到空间对象。
否则只是感知，不是系统。

3.语义地图不是给机器人看的，是给业务系统看的。
机器人只需要局部导航；
但系统需要知道“空间里发生了什么、归属于哪里、该怎么调度”。

## 本周练习题

练习1：增加 ZoneBlocked
要求：
在 models.py 增加一个事件类型
target_type=ZONE
触发后生成对该区所有 Lane 的巡检任务
这会逼你理解“区域级事件如何下钻为通道级任务”。

练习2：增加车位类型
在 Slot 里把 slot_type 用起来，例如：
NORMAL
VIP
CHARGING
再写一个查询函数：
get_free_charging_slots_by_zone(zone_id)
这会让你理解“语义层不是只有ID，还有业务属性”。

练习3：给 Lane 加方向约束
例如：
FORWARD
REVERSE
BIDIRECTIONAL
然后打印地图时把方向也展示出来。
这是在为未来路径规划做准备。

练习4：写一个“对象关系问答”
自己加一个简单函数：
输入 Slot-A01
输出：
属于哪个Lane
属于哪个Zone
当前状态是什么
这一步非常像以后 Agent 的工具函数。
