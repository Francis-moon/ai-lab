# FastAPI 接口层

目标不是做漂亮，而是做出一个最小可运行的 “停车空间状态 API”，为后面第10周 LangGraph 和第11周调度器做底座。
这套模板对应的其实就是你们未来系统里的最基础一层：
对象状态可查询、可修改、可被任务系统调用。

## 本周你要学会什么

第9周只解决 4 件事：
1,用 FastAPI 暴露 HTTP 接口
2,让数据库里的 Slot 可以被查询
3,让 Slot 状态可以被修改
4,理解“接口层 = 系统对外的标准入口”

你要把它理解成：
第7周：定义空间对象
第8周：把对象放进数据库
第9周：把数据库能力开放成 API

## 如何运行

第一步：初始化测试数据
在项目根目录执行：
python -m app.seed
你会看到：
Seed data inserted successfully.

第二步：启动服务
uvicorn app.main:app --reload
正常会看到：
Uvicorn running on http://127.0.0.1:8000

第三步：打开接口文档
浏览器访问：
http://127.0.0.1:8000/docs
这个页面非常重要。
FastAPI 最大优点之一就是自动生成 API 文档，你可以直接点按钮测试接口。

## 你本周必须完成的练习步骤

练习1：查看所有车位
浏览器打开：
http://127.0.0.1:8000/slots
你应该能看到所有 Slot 数据。

练习2：按区域筛选
访问：
http://127.0.0.1:8000/slots?zone=A
目标：只看到 A 区车位。

练习3：按状态筛选
访问：
http://127.0.0.1:8000/slots?state=free
目标：只看到空闲车位。

练习4：查询单个车位
访问：
http://127.0.0.1:8000/slots/A-001
目标：看到 A-001 的详细状态。

练习5：新增一个车位
在 /docs 页面里找到 POST /slots，输入：
{
  "slot_id": "D-001",
  "zone": "D",
  "state": "free"
}
执行后应成功创建。

练习6：修改车位状态
在 /docs 页面里找到 PUT /slots/{slot_id}/state
参数：
slot_id = D-001
Body：
{
  "state": "occupied"
}
执行后，D-001 状态会改成 occupied。

练习7：删除车位
测试 DELETE /slots/{slot_id}
删除：
D-001

## 本周核心点

下面这部分比代码更重要。

### 核心点1：FastAPI不是“网页框架”，而是“系统接口层”

你现在做的不是一个网站。
你做的是：
数据系统对外提供标准能力的入口
以后无论是：
LangGraph Agent
调度器
机器人
大模型
前端大屏
都不会直接改数据库，而是应该先打 API。
所以这一周你是在搭“总线入口”。

### 核心点2：接口设计要围绕“对象”而不是“页面”

你这里暴露的是：
/slots
/slots/{slot_id}
/slots/{slot_id}/state
这说明你的系统已经开始围绕 空间对象 Slot 设计，而不是围绕“一个页面点什么按钮”。
这是从“软件页面思维”走向“系统对象思维”的关键一步。

### 核心点3：状态更新必须有边界

你在代码里做了这个限制：
allowed_states = {"free", "occupied", "illegal", "blocked"}
这非常重要。
因为以后真正的 Spatial OS，不是随便写状态，而是：
状态必须标准化
事件必须标准化
任务必须标准化
否则系统会很快失控。

### 核心点4：API是后续 Agent 化的基础

第10周做 LangGraph 时，你不会让 Agent 直接改 Python 变量。
而是让 Agent 去调用这些 API，比如：
查询某区域空闲车位
修改某个 Slot 状态
根据状态生成任务
也就是说，这周代码未来会被 Agent 当成“工具”。

## 本周完成后的验收标准

你可以拿下面 5 条自测：
你能启动 FastAPI 服务
你能在 /docs 里测试接口
你能查某个 Slot
你能改 Slot 状态
你理解 API 是后续 Agent / 调度器调用的工具层
只要这 5 条成立，第9周就算过关。
